Hubzoid · Access Management

How access works

The finalized model. One convention does the work: a tool lives in a file, the file name is the permission, and an OpenWebUI group of the same name is the key. Everything else is enforcement around that.

The model

A file is a door. Its name is the permission. The matching OpenWebUI group is the key.

A critical tool lives in a file inside restricted/. The file name is the permission name. An OpenWebUI group of the same name unlocks every tool in that file. Nothing in restricted/ loads for a person who is not in the matching group.

restricted/sales.py  →  group sales   →  unlocks every tool in sales.py
restricted/finance.py  →  group finance  →  unlocks every tool in finance.py

The match is case insensitive and whitespace trimmed, so Sales, sales, and " sales " all map to the same door. A casing or spacing slip never silently locks someone out.

Hubzoid, once at setup

Set up the doors

  • One file per door. Name it by what it should mean to the client: sales, finance, inventory. The file name is yours, so friendly names are free.
  • Seed matching groups. We create OpenWebUI groups with the same names. The group list the client sees is exactly the list of permissions available.
  • A new door later is a deploy. A new system, or a narrower view of an old one, means a new file plus its group. The door has to exist before a key can open it.
Client admin, any time

Run it day to day

Add a person and grant access. Open OpenWebUI, add Meera, put her in the groups she needs: sales, finance, or both. Group membership is the grant. Two groups means both.

Open a new door. Create a group whose name matches a file. Matching the name is the whole link. Nothing else to configure.

No roles, and no separate grant screen. OpenWebUI's own group management is the admin tool. The client never logs into anything else.

Fixed vs editable. Hubzoid fixes the files at deploy, and therefore the permissions. The client edits the groups and who is in them, live in OpenWebUI, with no developer. So who has access is always the client's to change, and what doors exist is always a deploy.
How it is enforced

Two layers, and the model is never the gate

Authorization runs in code, before any tool executes. The model is never asked for permission. A jailbreak cannot grant what was not granted.

1. The offered list is filtered per person. Each session is built from the logged in user's groups, so the agent is handed only the tools that person may use. An ungranted tool is not refused, it is simply not there. No wasted turn, no confusing denial, and the names of sensitive tools are never shown to people who cannot use them.

2. Every restricted call is re checked at dispatch, and fails closed. If a call reaches the dispatch anyway, through a prompt injection naming a tool or a quirk of the agent SDK, the check denies it and writes the attempt to the audit log. The filtered list is the clean path. This is the wall.

The check lives in the shared tool dispatch, so the Claude Agent SDK and the OpenAI Agents SDK are covered by one rule. Every allow and every deny is recorded, which the audit log makes readable.

Audit log

Every decision, written where the decision is made

The allow or deny happens in the runtime dispatch, so the log is written there, not in OpenWebUI, which never sees a tool call. One append per decision: {ts, user, surface, tool, decision, reason}. Read it with hubzoid audit <hub>. This is the one view OpenWebUI does not give you.

One file does not grow forever. Logs are date partitioned, one file per month: logs/access-2026-06.jsonl. The writer picks the file from the date, so rotation needs no trigger and no extra dependency. hubzoid audit reads across the months and filters by date, person, or decision. Switch to daily files if a hub is high volume.

Retention is a policy knob: default keep twelve months, archive instead of delete for clients with compliance retention rules. Append only by design, so the trail is never rewritten. A searchable UI (self hosted Langfuse) is the later upgrade, not needed for this.

Surfaces

A restricted door needs a verified person behind it

The principal is who, on which surface. OpenWebUI carries a verified login, so it can reach restricted tools. Slack, Telegram, and scheduled runs carry no per person identity, so they get the non restricted set only. A restricted door is never reachable from Slack.

If one specific Slack user truly needs a restricted tool, map that single user to an identity and group by hand. Do not build general per user access on Slack.

Secrets

Credentials live where the model cannot read them

Secrets are never written into tool code, so they are never in git and never greppable. They live in restricted/ (for example restricted/.env) and load into the process environment at startup.

The file reading tools, read_file and list_files and grep, refuse any path under restricted/. This is a resolved path check in code, backed by a test that proves read_file("restricted/.env") is refused. A system instruction also tells the agent to stay out of the folder. The code block is the control. The instruction is the reminder.

A restricted tool's own code still reads the credential to do its work. The model only ever receives the result rows, never the secret. Outbound HTTP is held to an allowlist, so even a leaked string has nowhere to go.

Stronger option, for a crown jewel like an SSH key into a client's production system: hold it in a separate gateway process under a different operating system user, so the kernel, not a denylist, keeps the agent out. Optional, decided per secret.

Rows and columns · the data layer

The second axis files cannot express

A file and its group decide which tool a person may call and which columns everyone in that group shares. They cannot decide which rows each person sees, because a file is the same for everyone in its group while rows differ per person. A branch manager who may see only their own store is this axis.

It is enforced at the data layer from the verified identity, never in files or groups:

the person carries an attribute, for example store = BLR-01, held as user metadata.
the bridge passes that identity into the tool as a verified claim. The model never supplies it.
where the source system has its own access model, ride it (for example field privilege groups). Where it does not, impose a database floor.
the database filters every query by that scope (SQL Server row level security on the session context). Columns are limited by GRANT or masking.
one file, one group, but two people calling the same tool get different rows, enforced by the database.

The rule: the file decides the door and the shared columns. The database decides each person's rows. This is needed on day one wherever the client is multi branch.

finalizedOne foundation underpins all of this: the logged in identity and groups must reach the runtime as a verified claim. Tool access rides the file and group convention. Row and column scope rides the data layer. Secrets live outside the model's reach.
