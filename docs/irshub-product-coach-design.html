IRS Hub · Design Spec

Product Sense Coach & the Business-to-Dev Wall

A new module on the existing IRS Hub (Hubzoid) that coaches PMs through structured product reasoning — build vs. extend vs. reuse — and turns the result into a signed, filtered development-request ticket. Its real job: act as a product strategist that sits between the business/PM team and developers, so only well-reasoned requests get through.

01  Objective, principles & evidence

Add a coaching capability to IRS Hub that, given a requirement or initiative, guides a PM through structured product reasoning to decide whether to build new, extend existing, or reuse a capability — and to articulate the expected outcome and a rough cost-vs-benefit view. It then emits a structured brief and, on the PM's say-so, a development-request ticket. The coach asks the right questions and nudges; it never issues a build / no-build verdict.

The deeper goal is organisational: a wall between the business/PM team and the development team. The coach is the product-strategist front door, so the only requests that reach developers are evidence-backed, reuse-checked, and outcome-anchored.

  • Coach, not oracle.Never opens with a recommendation. Nudges through questions; ends by reflecting the PM's own reasoning back as a brief plus open risks. Optimises for building judgement, not replacing it.
  • Derive the capability map.Ground "could you reuse/extend what you already have?" in the actual product codebases, and ground outcome reasoning in vision/KPI docs.
  • Outcome before build.Every session surfaces the target metric and its current baseline before any build discussion. A missing baseline is flagged, never invented.
  • The wall filters; it does not gatekeep arbitrarily.What passes is a signed brief that survived all gates. The signature — not a human's mood — is what unlocks a ticket.
  • Necessity & deflection before a ticket.Establish whether a request is needed to launch (must-have) or merely nice-to-have, and whether an existing self-serve capability already covers it. Nice-to-haves are parked; config / how-to is deflected to self-serve; only launch-relevant, non-reusable needs become dev tickets.

Why now — what the board and the code already show

This is not a hypothetical risk. A read of the live business-request board (every ticket, last six months) and the matching code commits shows the exact failure mode the wall is built to stop: the platform is barely live — only a small share of programmes run on it and new programmes are not going live — yet the request queue is dominated by internal convenience, not launch-critical work.

~15% must-have to launch

Only ~15% of business-board tickets are genuinely required to put a programme live or let a participant register & pay. ~63% are nice-to-have, ~22% not-needed / edge. Of the open backlog, ~10 of ~110 are real launch-blockers — buried under ~100 non-essential requests competing for the same engineers.

~68% serve internal teams

By audience, ~68% of requests serve operators / finance / platform; only ~32% the participant. The single biggest category is operator preference; only ~19% are real core defects. Code agrees — roughly twice as much commit effort lands on internal / admin surfaces as on the participant experience, and that gap is widening.

The cost is tech-lead time

The technical leads each personally touched ~30 business tickets — ~85% non-launch and roughly half BA-handleable (config / data / preference) — on top of already-full coding loads. The most expensive engineers are being used as a triage queue while the few real launch-blockers wait.

Fragmentation of the clean, configurable platform is still contained (only a low single-digit share of requests actively risk it, and explicit "rebuild the old system" parity is small) — but it is trending the wrong way, and most of those fragmenting requests sit unnoticed inside the "nice-to-have" pile. A gate is what keeps "contained" from becoming "characteristic."

The deflect target already exists

Roughly half the queue is BA-handleable — config, policy, template, data and how-to requests that need no developer. The IRS Hub already offers these as self-serve (Template Builder, Config Helper, Support Agent). So the coach's first job is often not to write a better ticket but to deflect the request into an existing self-serve capability — no dev ticket created at all.

Caveat · used wrong, this makes it worse

An intake agent without a hard gate is a faster fragmentation engine: ask it for a ticket and it will happily produce a plausible-looking one, so volume rises. It only helps if it can say no / defer / "do it yourself," never creates a ticket that hasn't passed the necessity + reuse check, and is judged by deflection rate and new programmes launched — not tickets created. Build the gate first, the ticket-writer second.

02  Feasibility verdict

Yes — it's buildable as a module inside the existing IRS Hub, with ClickUp as the ticket destination. Two realities shaped the entire design.

Reality 1 · No internal ticketing to reuse

"PRS" is the Program Registration System — the event-registration platform itself (Litestar prs-facade + Odoo prs-backend + Celery prs-worker + React prs-frontend), not a product-request tool. Its "request" APIs create participant registrations. So development tickets go to ClickUp.

Reality 2 · No per-tool access control

Hubzoid has no in-hub RBAC. A sub-agent's tools: whitelist is discarded on load and the single agent gets the full tool registry. So the wall cannot be a tool-visibility boundary in v1 — it is a code-validated choke point plus a dev-team SLA, instrumented to harden into the access-management L4/L5 gate later.

Everything else the spec asks for maps cleanly onto Hubzoid primitives: a sub-agent for the persona, skills for procedures, knowledge files for the capability map and vision docs, a couple of Python tools for the deterministic spine, the ClickUp MCP connector for the write, and two scheduled jobs for refresh + metrics.

03  Three framework truths that decided the architecture read from source

The design was re-grounded against the Hubzoid source rather than assumptions. Three findings overturned the "obvious" approach:

① Guardrails must live inside a tool at call time, not in tool visibility

In hubzoid/loaders/agents.py, promote_to_skills discards a sub-agent's tools: whitelist (with a log warning); hubzoid/factory.py hands the single agent tools=list(registry.values()) — the whole registry. There is no per-agent tool scope. So the gate and the wall can't be enforced by hiding a "create ticket" tool. They are enforced by a deterministic validator that writes a signature, and a skill that refuses to file without it.

② tools_local Python cannot call MCP tools in-process

MCP servers attach to the Agent and are invoked by the model; @function_tool Python functions are a separate registry. A Python tool can't call clickup_create_task directly. Resolution: split the concern — Python does the deterministic part (validate, sign, dedupe-check, audit); the model makes the actual ClickUp write from a skill, but only after the signature exists in session state.

③ Session state needs a tool-owned file, not a hand-written artifact

write_artifact strips directory parts and is per-chat; chats are fresh-context each turn. A model re-reading a hand-written state.json is fragile (it may not know the chat-specific path). Resolution: a tools_local tool owns coach-state.json, resolving the per-chat dir itself via get_chat_id() + memory.chat_artifact_dir() — the exact mechanism the built-in artifact writer uses. State survives fresh context deterministically.

Why this matters

These three truths convert a naive "prompt-the-agent-nicely" design into one with a single deterministic choke point (the signature) and an honest account of what is and isn't enforceable today — which is exactly what makes the wall credible to a dev team and forward-compatible with your access-management rollout.

04  Architecture & module layout

One front-door sub-agent runs a 7-gate Socratic flow whose ledger lives in a code-enforced session-state machine, grounded in a grep-backed capability index and a vision/KPI context store, ending in a validated brief whose signature a deterministic check requires before a ClickUp ticket can be filed. All of it lives inside the existing IRSHub folder, beside the current Template Builder / Config Helper / Support Agent.

06  The coaching engine — seven gates, code-enforced

The gates run in order; you cannot skip to "build it." Integrity is not just prompt text — it is a state machine in coach_session.py. Every turn the sub-agent's first action is coach_session_state(get) to rehydrate, then it works only the lowest gate not yet closed-or-blocked.

  1. Problem & user — who hurts, in what job? close requires a stated problem + affected user.
  2. Evidence — real signals (data / tickets / interviews / observation), not anecdote. close requires ≥1 evidence item with a source; anecdote-only is labelled weak.
  3. Target outcome & baseline — which KPI moves, and what is it today? close requires baseline_value XOR baseline_missing=true. Pulled from vision/, else asked, else flagged — never invented.
  4. Existing-capability check — could config/extension of something you already have meet the goal? MUST call coach_capability_lookup, grep-verify the cited signal, and phrase levers as questions. close requires ≥1 cited lever OR an explicit no-lever record with a reason.
  5. Smallest viable test — the cheapest experiment that produces signal. close requires a described test + the signal it would produce.
  6. Cost & maintenance — build + ongoing + opportunity, as rough buckets with explicit assumptions. close requires each cost to carry an assumptions[] entry (no fabricated ROI).
  7. Synthesis — reflect the PM's reasoning back as the brief: build/extend/reuse options with rough cost + open risks. no verdict; runs brief_validate; emits the signature.
Why skipping is impossible

Defense in depth: (1) the prompt states the gates as a hard ordered ladder; (2) close_gate(N) refuses out-of-order or under-specified closes. The brief validator and ticket skill both read gate status from the same ledger, so a skipped gate cannot ship downstream.

Stop-and-name-the-gap

When the PM can't answer an early gate, the coach calls flag_gate(N, reason), stops advancing, and states the gap. Synthesis still runs in "gaps mode": the brief leads with the gap as the primary finding, validates as status=needs-info, and any ticket is tagged needs-info / baseline-missing — never ready-for-dev. The gap is a valid terminal output.

Deflect, don't ticket — and one question carries most of it

Gate 4 is also the deflection point: if the goal can be met by an existing self-serve capability (config, policy, template, data, how-to), the coach routes the PM there and no dev ticket is created. Synthesis additionally classifies each surviving request by launch-criticality (must-have / nice-to-have / not-needed) and audience (participant vs internal), so the queue is sortable by what actually launches programmes. The single question that does most of the filtering: "Does this put a new programme live, or let a participant register & pay? If not, why now?"

07  Capability Index — grep + knowledge files, no parser service

Three cooperating layers, deliberately pragmatic (honouring the "grep + knowledge files + anything else as needed" decision). No separate parser service.

A · Structured spine

capability-index.yaml — one entry per capability: id, product, kind (route|odoo_model|celery_task|fe_route|fe_api|config_flag), name, signal_path, signal_regex, summary, config_surface, extends_via, keywords[], source_repo, last_refreshed_sha. Hand-seeded for the top ~15–20 PRS capabilities, then maintained by the refresh job.

B · Deterministic query

capability_lookup.py ranks goal terms against the YAML and returns candidates with citations. Critical property: it can only return levers present in the file — the model cannot fabricate a lever. This is the structured value-add over raw grep.

C · Live verification

At gate 4 the coach runs grep_data with the candidate's signal_regex scoped to its signal_path (one repo at a time, under the 100-match / 25k-char caps) to confirm the lever still exists and quote the exact route/model/task — citing both the narrative file and the live line.

PRS offers strong machine-readable signals that make this high-yield: ~110 @get/@post route decorators in prs-facade Odoo _name= models across 20+ addons 14 Celery @app.task routers.ts / api.ts enums db-schema-reference.md

Honest limit

Grep catches decorators and model names but misses behaviour expressed in config, data, or Odoo dynamic dispatch. The coach may miss a lever or surface one that doesn't quite fit — which is exactly why every lever is cited (checkable) and phrased as a question (the PM corrects it), never asserted as fact. The weekly capability-refresh keeps the spine current; its commit pathspec is kept strictly to capabilities/ + the YAML so it never collides with the existing knowledge-refresh job.

08  Context Store — vision & KPI grounding

Plain markdown under knowledge/vision/, grounded via read_knowledge + grep_data (Hubzoid has no vector DB, and the corpus is small and human-curated, so RAG isn't needed). The IRS product lead drops north-star.md, okrs-current.md, and kpi-baselines.md (one metric per row: metric, baseline value, as-of date, source, owner) into the folder — the same mechanism as the existing 38 knowledge docs. For just-arrived material a PM can drag a doc into the chat and the coach reads it that session.

The no-docs reality (today)

No vision/KPI/OKR docs are wired into IRSHub yet. The module ships vision/ as templates with an INDEX.md banner: "NOT-YET-WIRED — no KPI baselines available; gate 3 will flag missing baselines as findings." Gate 3 then flags baseline_missing on most early sessions — which is correct behaviour, not a failure.

That gap is reframed as the headline output: the missing-KPI-source rate on the metrics dashboard becomes the measured business case for wiring one OKR doc in early. As real docs land, the same machinery resolves baselines and cites them with zero code change.

09  The Brief & its validator

Gate 7 produces two artifacts: brief.md (human, the PM reviews) and brief.json (machine, the ClickUp field source + signature input). brief_validate.py enforces the schema deterministically and, on pass, computes synthesis_signature = sha256(canonical brief + gate ledger) — the only thing that unlocks ticketing.

brief.json (shape, abridged)
{
  schema_version, session_id, pm_identity, created_at,
  status: "ready" | "needs-info",
  initiative_title,
  launch_criticality: "must_have | nice_to_have | not_needed",
  audience: "participant | operator | finance | platform",
  problem_and_user: { statement, who_is_affected, citations[] },
  evidence: [{ claim, type, source, strength: "strong|weak" }],
  target_outcome: { metric, baseline_value|null, baseline_missing, baseline_source,
                    target_value, owner },
  existing_lever_assessment: { levers:[{ capability_id, name, product,
                    kind:"config|extend|reuse|code", source_ref, fit, why }],
                    lever_identified },
  options: [{ approach:"reuse|extend|build_new", sketch,
              rough_cost:{build,ongoing,opportunity}, assumptions[] }],   # >=2 options
  smallest_viable_test: { description, signal_it_would_produce, effort },
  open_risks[], named_gaps[], gate_ledger[],
  provenance: { reflected_from, model_added_assumptions[], capabilities_cited[] },
  synthesis_signature,            # set by brief_validate on pass
  verdict: null                  # hard-nulled. there is no decision field.
}

Validation rules: every assertion-bearing field carries ≥1 citation; target_outcome has baseline_value XOR baseline_missing; options has ≥2 entries and no decision/verdict field; every rough_cost has a parallel assumptions[]; provenance.model_added_assumptions captures anything not traceable to the user, keeping the brief a faithful mirror.

10  Ticketing & the Wall

Destination & write path

ClickUp via the write-capable MCP server. One-time bootstrap (recorded in clickup-mapping.md as the single source of truth): a dedicated Folder "Product Requests (Coach)" → List "Coach Intake", pre-created space-level tags, and custom fields with pinned IDs. The actual clickup_create_task call is made by the model from file-coach-ticket (because Python can't call MCP in-process) — gated by the signature the Python validator produced.

Filing procedure

  1. Confirm session state holds a valid synthesis_signature — refuse otherwise, naming what's incomplete.
  2. Require the PM's explicit "file this." The coach never auto-files; the brief is a mirror first, a ticket second.
  3. Dedupe: clickup_filter_tasks on the Session Signature field — update instead of duplicate; surface near-duplicate open requests as a reuse question.
  4. clickup_create_task mapping brief.json → name, full brief as description, tags (coach-generated + ready-for-dev|needs-info + product + baseline-missing), and custom fields (Problem, Launch-Criticality, Audience, Target Metric, Baseline, Existing-Lever Verdict, Smallest Test, Rough Cost, Open Risks, Requesting PM, Coach Session ID, Session Signature).
  5. coach_audit_append the create/update; return the URL.

The wall — v1 process gate, instrumented

The IRS dev team agrees (in wall-policy.md + a pinned note on the list) to accept build/product work only from Coach Intake bearing coach-generated + a non-empty Session Signature. Anything hand-filed without those is bounced: "please run it through the coach." The hub's main agent routes raw build requests to the coach and refuses to file tickets itself.

Honest residual gap

Because every chat user sees every tool and there is no per-tool gating, a determined user could hand-call clickup_create_task mid-chat. A bare call has no valid signature, so the dev SLA + signature check catches it — but it is not fail-closed. True enforcement waits for the access-management L4 PreToolUse gate.

Forward-compatible — no rework

Every ticket + audit row already stamps identity (pm_identity → later the L3 bridge-validated principal), role via tags, audit (append-only JSONL = L7's record), and a named-operation shape (filing is one typed flow = the L5 pattern). When Hubzoid ships the L4 fail-closed hook, it gates clickup_create_task on "valid signature AND verified principal" — and the v1 fields are the schema it checks. The procedural wall becomes the policy wall with zero schema change.

11  Guardrails

GuardrailHow it's enforced
Never a verdictDoctrine forbids decision verbs; brief-schema.md has options[] and no decision field; brief_validate rejects any brief carrying a recommendation/verdict and hard-nulls verdict. Options are always a trade-off set; the smallest test is framed as an experiment, not a go-ahead.
Cite sourcesEvery capability claim carries capabilities/*.md + a live grep_data line; lookup only returns cited levers; a lever with no citation can't enter the assessment. KPI claims cite the vision/ filename. Validation fails on any uncited populated claim.
Flag missing KPI, don't inventGate 3 accepts baseline_value XOR baseline_missing=true — there is no code path to a number that wasn't read from a doc or supplied by the PM. A miss becomes an open risk + named gap + baseline-missing tag + a dashboard finding.
No fabricated ROICosts are rough buckets (S/M/L); validation refuses any cost without an assumptions[]; provenance.model_added_assumptions lists every figure the model introduced, keeping ROI labelled as assumption.
Pacing & faithful reflectionOne question (or ≤3 tightly-related) per turn; levers as questions; the brief's provenance block structurally separates user reasoning from model additions so reviewers can audit faithfulness.
The single choke point — and its honest limit

file-coach-ticket does a final read of session state before any write: missing/invalid signature, missing citation, non-null verdict, or a cost without an assumption → it refuses and returns the PM to the relevant gate. This is exactly where the future L4 hook attaches. Limit: the model still writes the prose, so subtle verdict-leaning or unlabelled assumptions can slip past a structural validator — provenance + rubric-based transcript review are the control, and that is human judgement, not proof.

12  Success metrics

All of these derive from data the system already writes — the coach-audit JSONL (written by coach_audit.py, since Hubzoid doesn't auto-log interactive side effects) plus ClickUp fields. coach-metrics-rollup.md rolls them weekly into a versioned knowledge/coach-metrics.md.

MetricDefinition
1 · Reuse-over-build rateSessions where existing_lever_assessment.fit ∈ {meets, partial} and the leading option ≠ build_new, over total synthesized sessions. Cross-checkable via the ClickUp Existing-Lever Verdict field.
2 · Outcome-with-baseline rateSessions reaching a target metric with a real baseline (not flagged), over total. Honest by construction because gate 3 forces the flag.
3 · Bypass reduction (the wall's KPI)Count of ClickUp tasks in dev lists lacking the coach-generated tag, trended week-over-week against coach volume. The dedicated list + signature is what makes bypass countable.
4 · PM growthEach session's coach-state.json (gate answers + citations + gaps) is the transcript corpus, linked from the ticket via Coach Session ID. The rollup samples N per PM; a human scores depth, evidence quality, and lever-engagement against coach-rubric.md over time.
5 · Deflection rateRequests resolved via an existing self-serve capability or parked as nice-to-have without a dev ticket, over total intake. The primary signal that the wall is removing load rather than reformatting it — the metric that distinguishes a gate from a faster ticket-writer.
6 · Launch-criticality of promoted ticketsShare of tickets that do reach the dev board classified must-have-to-launch. Rising = the wall is letting through what launches programmes. North-star pairing: new programmes gone live.

Plus a leading indicator the dashboard surfaces explicitly: the missing-KPI-source finding rate — the measured case for wiring KPI docs in.

13  Build order

Full spec is the goal; this internal order makes each layer independently testable, delivers a usable coach before any ClickUp wiring exists, and isolates the only write side-effect to last. The integrity spine is built before anything can write.

1

Contracts + gate machine

Write coach-doctrine.md, coach-rubric.md, brief-schema.md, then coach_session.py with the ledger, ordering enforcement, per-gate shape checks, and per-chat persistence. Unit-test gate-skip refusal and fresh-context reload.

→ a provably un-skippable state machine, no integrations
2

The coach persona

Write agents/product-coach/AGENTS.md + run-product-coaching, wired to session state only. Run a full 7-gate session via hubzoid run against a hand-written capability description; verify never-verdict and stop-and-name-the-gap. Pilot with 2–3 friendly PMs; tune pacing.

→ a coach that produces a brief end-to-end with zero external deps
3

Capability grounding

Hand-seed capability-index.yaml + capabilities/*.md for the top ~15–20 PRS capabilities, add capability_lookup.py, write gate-capability-check, wire gate 4 to lookup + grep verification. Add vision/ templates + INDEX stub so gate 3 degrades honestly day one.

→ the coach surfaces real, cited reuse levers
4

Brief generator + validator + audit

Write synthesize-brief + brief_validate.py (the signature) + coach_audit.py. Metrics begin accruing even before tickets exist.

→ validated, signed, downloadable briefs + the audit corpus
5

Ticket + wall

Edit .mcp.json for ClickUp; run the one-time bootstrap (folder/list/tags/fields), pin IDs into clickup-mapping.md; write file-coach-ticket with refuse-without-signature, dedupe, field map, audit. Write wall-policy.md and get the dev-team SLA in writing. Route build requests in the main AGENTS.md.

→ requirement → coached session → signed brief → tagged ClickUp ticket; wall live
6

Durability + instrumentation

Write capability-refresh.md (narrow commit pathspec) and coach-metrics-rollup.md; dry-run then bounded real run. Confirm the 4 metrics report. Run the standard HubzoidTests install-to-UI walkthrough before release.

→ self-maintaining capability map + a live metrics dashboard
