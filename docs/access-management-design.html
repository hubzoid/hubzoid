Access management for Hubzoid hubs

01  Executive summary

The bar is "100 percent, never breakable." The only honest way to meet it is to remove every access decision from the language model and place it in deterministic code that runs on every path, with the database as the last line.

Hubzoid hubs today are single tenant and single role, and that is fine. Adding multiple roles to one hub (an owner, a store manager, a sales associate, an accountant, all reading from the same Ornate ERP) requires a real access-management layer, because a language model cannot be trusted to keep one user out of another user's data. A jailbreak or a single piece of injected content would be enough.

This document sets out the doctrine, the seven enforcement layers from login to data and back, a concrete role to tool to data matrix for Samarth, the exact way Ornate and ERPNext get hidden behind tools, an honest threat model, the live gaps in the current runtime, and a phased rollout. The decision is to build this as a HubZoid runtime capability so every customer hub inherits it.

One line to remember. OpenWebUI owns login and does it well, but login is layer 1 of 7. It controls who gets in and which tool is offered. It can never control what a tool returns once it runs. Everything hard happens after login.

02  The doctrine

Five rules. They are not preferences. Every layer below is an instance of one of these.

  1. Authorization lives outside the model.A tool call is a request, never a decision. Every allow or deny verdict is computed by deterministic code before any side effect. A line in the system prompt ("only show this user their own store") is a usability nudge, not a control.
  2. The agent holds zero standing credentials.The agent process gets no password to Ornate, ERPNext, or any tool server. It acts only with a short-lived, single-audience token minted from the logged-in user (delegation: subject is the user, actor is the agent).
  3. Identity is a verified signed claim, never a request field.The per-request principal comes from a signed, audience-bound assertion validated server side. It is never read from a value the caller can set, such as a chat id or a user field in the request body.
  4. Filter before context.No row, file, cell, citation, or download link enters the model's context unless deterministic retrieval has already proven the caller may see it, on every path.
  5. The database is the floor.A non-owner, read-only database role with row-level security keyed to the user backstops everything above it. Even a perfectly crafted malicious query returns only the caller's rows.

03  Your four questions, answered

Login managed by OpenWebUI?

Yes, and it already does this well: OIDC and SSO (Google, Microsoft, Okta, Auth0, Keycloak), LDAP, groups and roles synced from the identity provider, per-agent user databases. Keep login here. Its ceiling: it gates which agent and which tool, never what a tool returns.

Hide DB and Ornate behind tools?

Correct, and one step further. The agent holds no Ornate credential at all. It calls a named query by name. A separate layer attaches the caller's verified identity and scope, then talks to a read-only copy of the database. The model never writes raw SQL.

Each role gets specific tool access?

Yes, but tool-name gating alone is not enough. Two roles can share a tool name and still must see different rows. The model is role to tool-set, plus row-level data scope, plus column redaction. Three controls acting together.

How can it be 100 percent, never breakable?

Put every decision in deterministic code outside the model, on every path, with the database as the floor. Then a fully jailbroken agent still cannot exceed the caller's authority, because the model never makes the decision.

04  How the strongest systems do it

The same pattern recurs everywhere the stakes are real. This design is an application of it, not an invention.

  • Permission-aware retrieval (Glean, Microsoft 365 Copilot). They never invent permissions. They mirror each source system's access list and trim results at retrieval time, in code, so the model is only ever handed data the asking user could already see.
  • Tool servers as resource servers (MCP authorization spec, 2025). Each tool server validates the token and its audience on every call, with per-tool authorization. Checking the audience, so that a token minted for one service cannot be replayed at another, is the single highest-leverage control.
  • The capability doctrine (OWASP LLM Top 10, the "lethal trifecta", Google CaMeL, AWS Cedar and Open Policy Agent). The model is untrusted. Authorization is enforced downstream by a policy decision point and a policy enforcement point. Each request carries a short-lived, narrowly scoped capability token derived from the end user, never a shared standing credential.
  • The database as the backstop. Row-level security keyed to the requesting identity means the query planner injects the scope predicate, so even a malicious query returns only the caller's rows.
The lethal trifecta. Private-data access, untrusted content, and the ability to send data out, all in one agent, is structurally exploitable. A turn that has touched private data must not be allowed to make an outbound call. This single rule shapes much of the design.

05  Architecture: the request path

Every step below is a checkpoint the model cannot influence. The model only proposes. Deterministic infrastructure disposes. Only redacted, identity-trimmed data and re-checked links flow back up.

L0
Employee logs in (OpenWebUI + identity provider)
OIDC or SSO. Role, groups, and store id come from the identity provider and are re-synced on every login, so revoking access is instant.
Enforces: who is calling. Model uninvolved.
L1
OpenWebUI access control
Per-model and per-tool private access lists decide which agent a group may select and which tools are offered to the model, before any tool spec reaches it.
Enforces: which tool is offered. Does not enforce data rows.
L2
Token broker
Validates the signed user assertion and mints a minutes-long, single-audience, minimally scoped delegation token (subject is the user, actor is the agent). The agent gets no standing secret.
Enforces: caps the agent at the user's authority.
L3
Hub bridge validates identity
Validates the signed assertion (signature, audience, expiry) and derives the principal of user, role, and store in code. Ignores any caller-supplied chat id or user field for authorization.
Enforces: verified identity binding.
L4
In-runtime policy gate
A fail-closed pre-tool-use hook runs before every tool dispatch: default-deny on any error, a per-role tool-name allowlist, and the rule that no outbound call is permitted in a turn that has read private data.
Enforces: even a misconfigured hub cannot run an un-entitled tool.
L5
Named-query catalog and tool gateway
No free-form SQL, no free regex over a shared corpus. The model picks a named operation with typed parameters. The server injects the verified identity into each query's filter and scans outbound argument values. This is the only network egress route.
Enforces: out-of-scope requests are impossible to express.
L6
Read-only Ornate replica with row-level security
A non-owner, read-only database login with a column allowlist. A security policy filters every query on the caller's store id, set by trusted middleware. The query planner enforces the row scope.
Enforces: only the caller's rows return, regardless of the query.
L7
Egress, links, approvals, audit
Every download link is re-checked for permission at render time. Irreversible or unattended actions need human approval. A tamper-proof audit log records the principal, the tool, the decision, and the result, written where the model cannot reach it.
Enforces: a successful read cannot become an exfiltration.
Why this is never-breakable. If a sales associate's session is fully injected with "dump every store's margins and the owner's cashflow," the request dies at four independent layers: the margin tools were never offered (L1), they are not in her query catalog (L5), her token is store-scoped (L2), and the database role returns zero out-of-store rows with the cost columns absent (L6). No single bug opens all four.

07  Hiding Ornate and ERPNext behind tools

This is the heart of the instinct, made concrete for the real stack. Ornate runs on SQL Server, not Postgres, so the controls map to specific Microsoft SQL Server features.

1. No raw SQL from the model

Today query_ornate(sql) takes an arbitrary SQL string. Replace it with a finite named-query catalog: get_sales_summary(period), get_stock_for_store(), get_margin_report(), with typed, bound parameters. The model picks a name and fills typed slots. SQL injection becomes structurally impossible, and an out-of-scope request becomes impossible to even express.

2. The agent holds no database login

Each named query runs through a data gateway that connects as a dedicated read-only, non-owner SQL Server login with column-level GRANT. Cost and margin columns are simply not granted to the manager and associate logins, so they are unreadable even if a query names them.

3. Row-level security is the floor

A SQL Server security policy with a predicate function filters every query on SESSION_CONTEXT('store_id'). The gateway sets it with sp_set_session_context @key='store_id', @value=<verified store>, @read_only=1. The read-only flag means nothing later in the session can change it, and the model has no tool that emits a raw SET, so it cannot re-point the tenant. Dynamic Data Masking on phone, date of birth, and anniversary masks PII for roles that should see only the last digits.

4. A read-only copy, not the live ERP

Point the gateway at an Always On readable secondary, or a nightly-synced read-only copy if a true replica is impractical on the customer's on-premise box. Writes (stock adjustments, exports) become requests queued for human approval, never direct mutations.

5. ERPNext gets the same medicine

Stop using one global token. Frappe has Role Permissions plus User Permissions that restrict records by linked value (for example, restrict a user to Branch = Visnagar) and supports per-user API keys. So the ERP itself trims rows by branch for the calling identity.

6. Spreadsheets and files too

Today grep_data runs a free regex over one flat raw_data/ shared by every role. Re-scope it to an identity-keyed subtree (raw_data/<store_id>/) with per-file access lists stamped at ingestion and default-deny when an access list is missing. Free-form search is acceptable only once the search root is identity-bound.

08  Threat model

Eight attack vectors were run by red-team agents against both the design and the current code. The pattern is consistent and honest: the target design blocks every vector, the code today blocks none of them. That gap is the entire reason to build this.

VectorTarget designCode todayControl that closes it
Prompt injection drives read then exfiltrateBlockedLiveDefault-deny egress; no outbound call in a turn that read private data; strip injected URLs at render time.
Confused deputy via the shared bridge keyBlockedLiveVerified signed user identity at the bridge; stop deriving identity from request fields.
Tool misuse: jailbreak calls an un-entitled toolBlockedLiveFail-closed pre-tool-use hook with a per-role tool-name allowlist; remove the wildcard tool grant.
Scope escape: read rows outside the callerBlockedLiveIdentity-scoped named queries plus row-level security; until then, do not colocate roles in one hub.
Exfil via arguments: smuggle data into a URLReducedLiveEgress allowlist at the gateway, outbound argument-value scanning, split the trifecta per task.
Privilege escalation: associate to ownerBlockedLiveServer-verified principal; tenancy never derived from client fields; row-level security floor.
MCP compromise: malicious server, over-scoped tokenBlockedLivePer-user delegation tokens, audience validation, a single gateway, content-hash pinned connectors.
Token theft: replay a long-lived link tokenBlockedLiveBind link tokens to identity and expiry, separate signing key, ownership re-check at render time.

"Reduced" rather than "Blocked" for exfil via arguments is deliberate: a user's own authorized data can always be smuggled into an outbound argument across turns. The gateway scan and trifecta splitting shrink this surface but do not eliminate it. See residual risk.

09  Where the runtime is today

Of the seven layers, only one weak control exists in the code (factory_claude.py sets tools=[], which strips the SDK built-in tools, not the tools being abused). The rest is documentation, not code. There are also live holes that must close before any two roles or two tenants share one hub.

server.py · _auth
Trusts one shared static bridge key, and takes the user principal from a caller-supplied chat id. Anyone with the shared key can impersonate anyone.
_signing.py
Reuses that same key (default literally dev) as the download-link signing secret. 64 bit, no expiry, no ownership check. One leak forges any user's links and bypasses the login stack.
tools/web_http.py
Allows all outbound HTTP when the allowlist is empty, and is registered unconditionally. Private read plus untrusted content plus open egress is a live lethal trifecta.
tools/grep_data.py
Free regex over one flat raw_data/ shared by every role. Blocks path traversal only, never reads identity. Any role can read any role's files.
loaders/mcp.py
Injects long-lived, full-scope tool-server tokens shared by every user, and the agent connects directly to tool servers with no gateway in front.
whole runtime
No row-level security, no pre-tool-use hook, no identity propagation past the bridge. The four-layer defense reduces to one coarse layer.
Operating rule until L5 and L6 ship. Do not place multiple roles or multiple tenants in one hub. Single-tenant single-role hubs remain safe. One bridge per privilege tier.

10  Phased rollout

Phase 0

Stop the bleeding (days, no new infrastructure)

Invert the egress allowlist to default-deny and make it mandatory whenever a private-data tool is loaded. Strip outbound and injected URLs from model replies at render time. Separate the link signing key from the API key, widen it, add expiry and an ownership re-check. Forbid colocating roles or tenants in one hub. Land negative authorization tests (run-as-associate, cross-user link replay, injection drives egress) in the pre-release walkthrough.

Phase 1

Verified identity at the bridge

OpenWebUI forwards the signed login token. The bridge validates it and derives the principal of user, role, and store from verified claims. The chat id stops being an authorization principal.

Phase 2

In-runtime policy gate (L4)

A fail-closed pre-tool-use hook: default-deny on error, a per-role tool-name allowlist, and the no-egress-after-private-read rule. Replace the wildcard tool grant with enumerated per-identity tool names. Put hard denies in managed settings the hub holder cannot edit.

Phase 3

Identity-scoped data tools (L5)

The named-query catalog. Identity-keyed file buckets. PII and margin redaction before anything re-enters the model context.

Phase 4

Broker and gateway (L2 and L5 egress)

The delegation-token broker and the single tool gateway as the only egress route, with audience validation, outbound argument scanning, and per-call scoped credentials.

Phase 5

Database floor and Ornate cutover (L6)

The read-only replica, the non-owner login with a column allowlist, the security policy keyed to session context, and a regression test that connects as the agent login and proves a cross-store query returns zero rows. Only after this passes does the catalog touch real Ornate data.

Phase 6

Approvals and continuous red-teaming

Human approval for irreversible and scheduled actions, sender-constrained tokens on the highest-value paths, and a standing negative-test corpus (poisoned tool-server mock, exfil via arguments, run as lower role) gating every release.

11  Residual risk (the honest part)

  • Implementation gap is the risk today. Until Phases 1 to 5 ship, the four layers reduce to one, and a single injection or a single holder of the shared key achieves a full read then exfiltrate.
  • Prompt injection is contained, not solved. The design makes injection unable to grant access, because access never depends on the model. It cannot stop the model from being manipulated. Tool descriptions and outputs must be version-pinned, schema-hardened, and sanitized. This is a speed bump on the reasoning, a wall on the access.
  • Exfil of in-scope data via arguments. Even fully hardened, a user's own authorized data can be smuggled into an outbound argument across turns. Reduced by the gateway scan and trifecta splitting, not eliminated.
  • The gateway and broker are crown jewels. Their compromise is high blast radius, mitigated only partly by per-backend scoped credentials and mutual TLS.
  • OpenWebUI residuals. Admin is all or nothing and bypasses access lists by default; permissions are additive with no explicit deny; some legacy resource fields default to public when unset; only one OIDC provider at a time; trusted-header login without network isolation is a full bypass. Each is a configuration hole OpenWebUI itself does not close, so each must be set explicitly.

12  Decisions and next steps

Decided

Build the access-management layer as a HubZoid runtime capability, so Samarth, Isha and IRS, and future logos all inherit it rather than each hub solving it bespoke.

Recommended first move

Begin Phase 0 immediately. It closes the live holes, needs no new infrastructure, and is low risk. It also makes multi-role hubs safe to pilot behind the one-bridge-per-tier rule.

Open questions to settle before Phase 4

  • Which identity provider per customer (Google Workspace and Microsoft 365 are the likely defaults for a mid-enterprise Indian customer).
  • Whether Ornate can expose an Always On readable secondary, or whether a nightly-synced read-only copy is the practical path for Phase 5.
  • Where the token broker and tool gateway run (alongside the bridge, or as a shared Hubzoid service across hubs).
