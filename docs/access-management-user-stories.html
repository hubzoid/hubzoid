Hubzoid · Access Management

User stories

Draft for review. How access is set up once, and how a client manages it day to day. Names do the work: tool name, permission name, and OpenWebUI group name are the same word.

Hubzoid, once at setup

Restricted tools define the permissions

Critical tools live in a restricted/ folder and are gated by default. Each tool's name is its permission. The list of restricted tools is the list of permissions.

restricted/ornate.py  →  permission ornate  →  OWUI group ornate
restricted/erp.py     →  permission erp    →  OWUI group erp

At setup we create OWUI groups with the same names. So the groups an admin sees in OpenWebUI are exactly the permissions available. Adding a new system later (for example Tally) is a Hubzoid deploy, because the door has to exist before a key can open it.

Client admin, any time

Story 1: Add a person and grant access

I open OpenWebUI, add Meera, and put her in the groups that match what she should reach: ornate, erp, or both. Adding her to a group is how I grant the permission. A person in two groups has both.

It does not matter whether you think of it as a group, a role, or a direct permission. With matching names they are the same thing.

Client admin, any time

Story 2: Create the groups

Permissions already exist as the restricted tools. I just create OpenWebUI groups with matching names (or they are seeded for me at setup) and assign people. Matching the name is the whole link. Nothing else to configure.

Optional: if I want one friendly role that bundles several permissions (a sales role that means erp plus something), that one case needs a small mapping and a tiny screen. Not needed to start.

The OpenWebUI question

How the admin selects the permission

Entirely inside OpenWebUI's existing group management. There is no extra screen for the simple version.

  • Grant a permission = add the user to the group of that name.
  • See the available permissions = the OWUI groups list, seeded to match the restricted tools.
  • A role bundling permissions = the only case that needs a small mapping. Optional.
What is fixed vs editable: the list of tools and permissions is set by Hubzoid at deploy. Groups, who is in them, and therefore who has access are live, client-editable in OpenWebUI, with no developer.
